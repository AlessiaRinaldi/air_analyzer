Firmware

This section describes the embedded firmware of the Air Analyzer system.

The firmware is designed to acquire the pressure signal generated by breathing inside the mask, store the raw samples locally, and reduce CPU activity during idle periods. The current implementation follows a simple low-power, interrupt-driven architecture based on STM32CubeMX, STM32 HAL, SPI, and DMA.

Firmware Overview

The firmware is built around three main ideas:

  1. configure the pressure sensor once at startup;
  2. keep the main program inactive during idle time;
  3. handle acquisition and storage only when an event occurs.

This approach avoids continuous CPU polling. The microcontroller remains mostly inactive and wakes only when data must be moved from the pressure sensor to the external memory.

The current firmware acquires only the pressure signal. Temperature data are not acquired, because the respiratory waveform is extracted from pressure variations inside the mask.

Development Environment

The firmware was developed starting from an STM32CubeMX project.

CubeMX was used to generate the basic configuration for:

  • system clock;
  • GPIO pins;
  • SPI communication;
  • DMA support;
  • USART interface for debugging;
  • USB interface for post-session communication.

The application-specific code is added inside the user-code sections generated by CubeMX, so that the project can be regenerated without overwriting the custom firmware logic.

Project Structure

The firmware is located mainly in the following project folder:

micro_usb/

The main application logic is located in:

micro_usb/Core/Src/main.c

The configuration and peripheral initialization files are generated by CubeMX and organized in the standard STM32 project structure.

Firmware Architecture

The firmware can be divided into three main parts:

Startup and initialization
          ↓
Low-power main loop
          ↓
Interrupt-driven acquisition and storage

The initialization code is executed only once when the device starts. After that, the firmware enters a cyclic acquisition workflow in which SPI transfers are performed using DMA and completed through interrupts.

Initialization Phase

At startup, the microcontroller initializes the peripherals configured in STM32CubeMX. This includes the clock tree, GPIO pins, SPI interface, DMA controller, USB interface, and USART interface.

After the peripheral setup, the pressure sensor is configured for continuous acquisition at 10 Hz. This sampling frequency was chosen because it provides a large margin with respect to the expected respiratory frequency during physical activity, while keeping memory usage limited.

Only the pressure output of the LPS22HBTR is acquired. The pressure value is provided by the sensor as a 24-bit raw sample, so each acquired pressure sample occupies exactly 3 bytes.

Once the sensor has been configured, the first SPI acquisition is started. From this point on, the firmware operates in an event-driven way rather than continuously polling the sensor.

Low-Power Main Loop

The main loop is intentionally kept almost empty.

Instead of continuously checking whether new sensor data are available, the microcontroller waits for interrupts. During this waiting phase, the CPU can remain inactive, reducing unnecessary power consumption.

This behavior is important for a wearable battery-powered system. The firmware does not require the processor to be active during the entire recording session. The CPU only needs to wake up briefly when a transfer is completed and a new pressure sample must be handled.

The low-power strategy is therefore based on event-driven execution:

CPU waits
   ↓
DMA transfer completes
   ↓
Interrupt occurs
   ↓
Sample is stored
   ↓
CPU returns to waiting state

Interrupt-Driven Acquisition

The acquisition process is managed using SPI with DMA support.

DMA allows the SPI peripheral to transfer data without requiring the CPU to manually handle every byte. When the transfer is complete, an interrupt is generated and the firmware processes the acquired sample.

At each acquisition event, the firmware:

  1. receives the raw pressure sample from the LPS22HBTR;
  2. extracts the useful pressure bytes;
  3. prepares the data for storage;
  4. writes the sample to the external EEPROM;
  5. starts the next acquisition cycle.

This creates a cyclic acquisition pipeline in which the CPU is active only for short processing intervals.

Pressure Data Format

The LPS22HBTR pressure output is stored as a 24-bit raw value.

The firmware stores this raw value directly in EEPROM without converting it to floating-point pressure values during acquisition. This choice reduces computational overhead and avoids unnecessary processing on the microcontroller.

Each stored sample has the following size:

1 pressure sample = 3 bytes

The conversion to physical pressure values is performed later during offline analysis on a PC.

The pressure conversion is:

pressure_hPa = raw_pressure / 4096

Storing raw samples is convenient because it preserves the full sensor resolution while minimizing memory usage.

External Memory Storage

The system uses an M95256 SPI EEPROM for local data storage.

The EEPROM capacity is:

256 Kbit = 32 KB = 32768 bytes

Since each pressure sample occupies 3 bytes and the sampling frequency is 10 Hz, the approximate recording time is:

32768 bytes / (3 bytes/sample × 10 samples/s) ≈ 1092 s ≈ 18 min

This storage strategy avoids continuous wireless transmission during the recording session. Data are saved locally and can be retrieved after the session through the communication interface.

Data Retrieval

During acquisition, the device does not continuously stream data to an external receiver.

The recorded pressure samples are stored locally in EEPROM. After the recording session, the user can connect the device to a PC and download the stored data for offline processing.

This design choice reduces power consumption because no Bluetooth or Wi-Fi transmission is required during exercise.

Offline Processing

The firmware does not compute the breathing frequency in real time.

Instead, it stores the pressure waveform in raw format. The respiratory signal can then be analyzed offline to extract:

  • inhalation and exhalation cycles;
  • breathing frequency;
  • pressure waveform shape;
  • differences between subjects or exercise conditions.

This separation between acquisition and analysis simplifies the embedded firmware and reduces the processing load on the microcontroller.

Low-Power Design Rationale

The firmware contributes to low-power operation through three main mechanisms.

Strategy Purpose
Local storage Avoids continuous wireless transmission
DMA-based SPI communication Reduces CPU involvement during data transfer
Interrupt-driven execution Keeps the CPU inactive during idle periods

Together, these choices allow the firmware to acquire respiratory pressure data while limiting active processor time.
